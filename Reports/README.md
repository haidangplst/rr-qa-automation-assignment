# Test Reports Directory

This folder contains all HTML test execution reports generated by the Playwright test automation suite.

## 📋 Report Organization

All test reports are automatically saved to this directory after test execution with the following naming convention:

### Naming Convention
**Format:** `report_{HH}_{dd}_{MM}_{yyyy}.html`

**Components:**
- `HH` - Hour (00-23, 24-hour format)
- `dd` - Day of month (01-31)
- `MM` - Month (01-12)
- `yyyy` - Full year (2026)

**Examples:**
- `report_09_16_04_2026.html` - April 16, 2026 at 9:00 AM
- `report_21_16_04_2026.html` - April 16, 2026 at 9:00 PM
- `report_14_25_12_2025.html` - December 25, 2025 at 2:00 PM

## 📂 Directory Structure

```
Reports/
├── report_21_16_04_2026.html
├── report_09_17_04_2026.html
├── report_14_18_04_2026.html
└── [other report files...]
```

## 📊 Report Contents

Each HTML report includes:

### 1. **Execution Summary**
- Total number of tests
- Number of passed tests
- Number of failed tests
- Pass rate percentage
- Total execution duration
- Visual progress bar

### 2. **Test Details**
- Individual test names and categories
- Status indicators (✓ PASSED / ✗ FAILED)
- Duration for each test
- Failure messages and details

### 3. **Environment Information**
- Operating System
- .NET version
- Test Framework (NUnit + Playwright)
- Browser used (Chromium)
- Execution mode (Parallel/Sequential)

### 4. **Analysis**
- Category-wise breakdown
- Key findings
- Failed test details
- Recommendations for improvements

## 🚀 Generating Reports

### Automatic Report Generation
Reports are automatically generated after running tests:

```bash
# Run tests and generate report
cd PlaywrightTests
dotnet test

# Or run specific test suite
dotnet test --filter "FullyQualifiedName~CategoryPagesTests"
```

### Manual Report Generation
If needed, you can manually generate a report using the report generator script:

```bash
# From project root directory
pwsh generate-report.ps1
```

## 🗑️ Managing Old Reports

### Retention Policy
By default, reports older than 30 days are candidates for cleanup. However, no automatic deletion occurs.

### Manual Cleanup

```bash
# Remove reports older than 30 days (Windows)
forfiles /S /D +30 /M "report_*.html" /C "cmd /c del @file"

# List all reports sorted by date
dir /OD report_*.html
```

### Archive Strategy

For long-term storage:

1. **Create Archive Folder:**
   ```
   Reports/
   ├── 2026-04/
   │   ├── report_09_15_04_2026.html
   │   ├── report_14_16_04_2026.html
   │   └── report_21_17_04_2026.html
   ├── 2026-05/
   │   └── ...
   └── [current reports]
   ```

2. **Compress Old Reports:**
   ```bash
   # Zip reports from specific month
   7z a "Reports_April_2026.zip" "2026-04/*.html"
   ```

## 🔍 Interpreting Report Metrics

| Metric | Meaning | Good Value |
|--------|---------|-----------|
| **Pass Rate** | Percentage of passing tests | 90%+ |
| **Total Tests** | Number of tests executed | Consistent count |
| **Duration** | Total execution time | Stable, no sudden increases |
| **Failures** | Number of failed tests | 0 (ideal) |

## 🛠️ Configuration

Report generation can be configured via `ReportConfig.json`:

- `enabled`: Enable/disable report generation
- `format`: Report file format (HTML)
- `outputDirectory`: Where to save reports (default: ./Reports)
- `fileNamePattern`: Report naming convention
- `autoRetentionDays`: Days before report cleanup (default: 30)
- `generateOnTestCompletion`: Auto-generate after tests
- `includeScreenshots`: Include failure screenshots
- `detailedFailureInfo`: Include detailed error messages

## 📝 Tips & Best Practices

✓ **Do:**
- Keep meaningful copies of important reports
- Archive reports monthly for compliance
- Use reports to identify flaky tests
- Share reports with team for visibility
- Track trends over multiple test runs

✗ **Don't:**
- Manually delete current day's reports
- Ignore failed test patterns
- Mix different test suite reports together
- Assume reports are backed up automatically
